---
title: FAQ
description: Frequently asked questions about the Python Port Scanner.
---

<AccordionGroup>
  <Accordion title="Do I need to install any packages?">
    No. The Python Port Scanner uses only the Python standard library — `argparse`, `socket`,
    `threading`, `json`, `csv`, and `datetime`. No `pip install` step is required.
  </Accordion>

  <Accordion title="What Python version is required?">
    Python 3.8 or later. The scanner does not use any features introduced after 3.8, so any
    reasonably modern Python installation will work.
  </Accordion>

  <Accordion title='Why are some ports showing as "filtered"?'>
    A port is marked `filtered` when the TCP connection attempt times out after 3 seconds with no
    response. This typically means one of the following:

    - A firewall is silently dropping packets destined for that port.
    - No service is listening and the host is configured not to send a TCP RST.
    - The network path to the host is unreliable or very slow.

    A `filtered` result does **not** mean the port is definitely closed — it means the scanner
    could not determine the state within the timeout window.
  </Accordion>

  <Accordion title="How do I scan all 65535 ports?">
    Pass `-s 1 -e 65535` to cover the full port range:

    ```bash
    python scanner.py 192.168.1.1 -s 1 -e 65535
    ```

    <Note>
      Scanning all ports takes significantly longer than the default range. With the default 3-second
      socket timeout, filtered ports alone can add hours to a full scan. Consider lowering
      `--threads` and `--rate-limit` to avoid overwhelming the target, and be prepared for the scan
      to run for an extended period.
    </Note>
  </Accordion>

  <Accordion title="Is it safe to use 200+ threads?">
    Using a very high thread count carries several risks:

    - **IDS/IPS alerts** — Rapid parallel connection attempts are a classic port-scan signature.
      Security systems on the target may block your IP or alert an administrator.
    - **Host overload** — Embedded devices, IoT hardware, or lightly resourced VMs may become
      unresponsive when flooded with simultaneous connection attempts.
    - **OS file descriptor limits** — Each thread opens a socket, which consumes a file descriptor.
      Linux systems default to 1024 open files per process. Exceeding this limit causes socket
      errors and unreliable results.

    The default of 100 threads is a reasonable balance between speed and safety. If you need faster
    scans, increase `--threads` incrementally and monitor for errors.
  </Accordion>

  <Accordion title="Why does the output file use a timestamp in the name?">
    When no `-o` flag is provided, the scanner auto-generates a filename such as
    `scan_results_2026-03-16_14-30-00.json`. The timestamp ensures that each run produces a unique
    filename, preventing results from a previous scan from being silently overwritten.

    If you prefer a fixed filename, pass it explicitly with `--output`.
  </Accordion>

  <Accordion title="Can I scan multiple hosts at once?">
    No. The scanner accepts a single `target` per invocation. To scan multiple hosts, call the
    script once per target, for example with a shell loop:

    ```bash
    for host in 192.168.1.1 192.168.1.2 192.168.1.3; do
        python scanner.py "$host" -o "${host//./_}.json"
    done
    ```

    <Warning>
      Automating scans across many hosts significantly amplifies the legal and ethical
      considerations. Only scan systems you own or have explicit written permission to test.
    </Warning>
  </Accordion>

  <Accordion title="Why does scanning take so long?">
    The dominant factor is the 3-second socket timeout applied to each port probe. For a port that
    is `filtered` (no response from the host or firewall), the scanner must wait the full 3 seconds
    before moving on. If many ports are filtered, the total wait time grows quickly.

    The timeout value is not configurable through CLI flags. To reduce total scan time:

    - Narrow the port range with `-s` and `-e` to focus on ports relevant to your goal.
    - Increase `--threads` so more ports are probed in parallel (within system limits).
    - Increase `--rate-limit` to dispatch probes faster.

    These adjustments reduce wall-clock time but do not change the per-port timeout.
  </Accordion>

  <Accordion title='The service name shows "Service name NOT FOUND" — what does that mean?'>
    The scanner resolves service names by looking up each port number in the operating system's
    services database (typically `/etc/services` on Linux and macOS). If the port number is not
    registered in that database, the lookup fails and the scanner reports `Service name NOT FOUND`.

    This result is purely informational — it reflects the absence of a well-known service
    assignment for that port number, and says nothing about whether the port is actually open or
    closed on the target.
  </Accordion>

  <Accordion title="Is this tool legal to use?">
    Port scanning is legal only on systems you own or on systems for which you have **explicit
    written permission** from the owner. Scanning systems without authorization may violate computer
    fraud and abuse laws in your jurisdiction, regardless of whether any harm is intended.

    <Warning>
      Always obtain permission before scanning any system you do not personally own and control.
      Unauthorized port scanning can result in legal consequences.
    </Warning>
  </Accordion>
</AccordionGroup>
