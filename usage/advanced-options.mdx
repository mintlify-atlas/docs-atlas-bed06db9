---
title: Advanced options
description: Tune thread count and rate limiting to control scan speed, stealth, and resource usage.
---

Beyond the target and port range, the scanner exposes two flags that significantly affect performance and behavior: `-t` for thread concurrency and `-r` for the probe rate limit. Combining them lets you tune the scanner for specific use cases.

<Warning>
Port scanning without explicit permission is illegal in many jurisdictions and violates most network acceptable-use policies. Only scan systems you own or have written authorization to test.
</Warning>

## Thread count (`-t` / `--threads`)

The `-t` flag sets the maximum number of concurrent threads used to probe ports simultaneously. The default is `100`.

```bash
python Port_scanner.py 192.168.1.1 -t 200
```

Higher thread counts reduce total scan time because more ports are probed in parallel. However, pushing this value too high can:

- Exhaust local file descriptor limits
- Trigger intrusion detection systems on the target
- Overwhelm the target host or intermediate network equipment

<Tip>
Start with the default of 100 threads. Increase to 200–500 only on stable, high-bandwidth local networks where you control both ends. For remote targets over the internet, values between 50–100 are generally safer.
</Tip>

## Rate limit (`-r` / `--rate-limit`)

The `-r` flag caps the number of port probes sent per second regardless of thread count. The default is `10` probes per second.

```bash
python Port_scanner.py 192.168.1.1 -r 50
```

Rate limiting decouples throughput from concurrency. You can have 200 threads active but still restrict probes to 5 per second — useful when you want to stay under IDS detection thresholds or avoid disrupting a production host.

| Use case | Recommended rate |
|---|---|
| Stealthy scan (evade IDS) | 1–5 probes/sec |
| Standard scan | 10–50 probes/sec |
| Fast scan (trusted local network) | 100+ probes/sec |

## Example: slow, stealthy full-port scan

This example scans all 65535 ports at a low rate and saves results to a CSV file. It is suitable for scenarios where staying under detection thresholds matters:

```bash
python Port_scanner.py target.com -s 1 -e 65535 -t 50 -r 5 -o full_scan.csv
```

- 50 threads keeps memory and connection overhead low
- 5 probes/sec reduces the chance of triggering rate-based alerts
- Scanning all 65535 ports at this rate takes considerable time — plan accordingly

## Example: fast scan of common ports

This example scans the standard 1–1024 range quickly on a trusted local network:

```bash
python Port_scanner.py 10.0.0.1 -s 1 -e 1024 -t 200 -r 100 -o fast.json
```

- 200 threads and 100 probes/sec are appropriate for a local LAN
- Results are saved directly to `fast.json`

## Combining flags: CTF example

Capture-the-flag environments are isolated and you control the target, so aggressive settings are acceptable:

<CodeGroup>

```bash Aggressive CTF scan
python Port_scanner.py 10.10.10.50 -s 1 -e 65535 -t 500 -r 500 -o ctf_results.json
```

```bash Targeted service scan
python Port_scanner.py 10.10.10.50 -s 8000 -e 9000 -t 100 -r 100 -o services.txt
```

</CodeGroup>

<Warning>
The high thread and rate values above are only appropriate for isolated CTF or lab environments. Do not use these settings against production systems or any host without explicit authorization.
</Warning>
